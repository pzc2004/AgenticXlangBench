## 概述

在一个 Python C 扩展中注入一个 GIL(全局解释器锁)释放错误。
Bug 在 C 层，症状在 Python 层表现为**多线程调用时偶发错误结果**，
**延迟不确定** —— 取决于线程调度时机(概率性触发)。

## Bug 设计

- **位置**：C 扩展的某个计算函数(C)
- **类型**：在修改共享状态前释放了 GIL(`Py_BEGIN_ALLOW_THREADS` 位置错误)
- **效果**：多线程并发时数据竞争，导致偶发错误结果
- **触发条件**：仅当多个线程同时调用该函数时；单线程完全正确

## 延迟显现机制

```
Python 多线程调用 C 扩展函数
    ↓ 线程 A 释放 GIL,开始计算
    ↓ 线程 B 获得 GIL,也进入计算
    ↓ 两个线程同时修改共享状态 → 数据竞争
结果:偶发错误(有时对,有时错)
```

**关键**：单线程 100% 正确。Agent 很难想到"多线程"是问题。
概率性触发使得复现本身就很难。

## 为什么难

1. **概率性触发**：不是每次多线程都错
2. **单线程完全正确**：agent 通常用单线程测试
3. **需要理解 CPython GIL 模型**：什么时候释放 GIL 是安全的
4. **需要并发调试**：ThreadSanitizer、helgrind 等工具
5. **跨语言**：Python 多线程 + C 扩展

## Anti-hack 措施

| Hack 路径 | 检测方法 |
|---|---|
| 强制单线程运行 | 多线程测试(必须并发) |
| 加锁掩盖(用 Python Lock) | 检查 Python 层加锁(应该是 C 层修) |
| 用 multiprocessing 替代 | 性能测试(进程开销 > 线程) |
| 只改 Python 不改 C | diff 只允许 .c/.h |
