# Task 10： Redis 模块 C 代码写越界 → 特定命令序列后数据损坏

## 概述

在一个 Redis 模块的 C 实现中注入一个内存写越界错误。
Bug 在 C 层，症状在 Redis 命令层表现为**特定命令序列后某个 key 的值变成乱码**，
**延迟 ~10 个命令才显现** —— 因为越界写破坏了相邻 key 的内存。

## Bug 设计

- **位置**：Redis 模块的 `RedisModule_StringDMA` 调用(C)
- **类型**：处理特定 key 类型时，内存写越界(少算了 1 个字节)
- **效果**：写入时越界 1 字节，破坏相邻 key 的内存
- **触发条件**：仅当 key 的 value 长度为特定值(如 16 字节对齐边界)时

## 延迟显现机制

```
HSET mykey field value  (value 长度恰好在对齐边界)
    ↓ 模块处理时写越界 1 字节,破坏相邻 key 的内存
后续 HSET 其他 key → 正常
    ↓ 累积 10 个命令后
HGET mykey field → 返回乱码(因为内存被后续写入覆盖)
```

**关键**：bug 在 `HSET` 时触发，但症状在后续 `HGET` 时才显现。
中间的命令都"正常"。

## 为什么难

1. **跨 2 层**：Redis 命令 → C 模块实现
2. **症状延迟**：bug 执行和症状出现之间有多个命令
3. **需要理解 Redis 模块 API**：`RedisModule_StringDMA` 的内存语义
4. **触发条件隐蔽**：特定 value 长度才触发
5. **需要复现特定命令序列**：顺序很重要

## 预估轨迹

- **步数**：200+ 步
- **时间**：2-3 小时
- **关键转折点**：
  - 步骤 30：发现 HGET 返回乱码
  - 步骤 80：检查 key 存储(看起来正常)
  - 步骤 120：对比不同操作序列，发现特定序列才触发
  - 步骤 160：怀疑模块代码
  - 步骤 200：找到 C 模块中的写越界

## Anti-hack 措施

| Hack 路径 | 检测方法 |
|---|---|
| 用原生 Redis 命令替代模块 | 检查是否绕过了模块 |
| 后处理结果(修复乱码) | grep 结果处理代码 |
| 重新 SET 正确值 | 检查是否修改了测试数据 |
| 只改配置不改 C | diff 只允许 .c/.h 文件 |
| 用 Lua 脚本替代模块 | 检查是否换了实现方式 |
